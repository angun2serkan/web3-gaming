# Task 0.6 — CI/CD Pipeline

**Faz:** 0 — Proje Altyapisi
**Tahmini Efor:** 1 gun
**Durum:** Beklemede
**Bagimlilik:** Task 0.1–0.5 (Monorepo, Frontend, Backend, Database, Contracts) — tamamlanmis olmali

---

## Amac

GitHub Actions ile otomatik lint, type-check, test ve build pipeline'i kurmak. Her push ve pull request'te kod kalitesi ve derleme butunlugu otomatik olarak dogrulanacak. Branch protection kurallarilya `main` branch'e sadece pipeline'dan gecen kodun merge edilmesi saglanacak.

Bu task tamamlandiginda:
- Her PR acildiginda ve her push'ta otomatik olarak lint, type-check ve build adimlari calisacak
- Frontend, backend ve contracts paketleri paralel olarak dogrulanacak
- PR'lar tum check'ler gecmeden merge edilemeyecek (branch protection)
- Turborepo remote cache ile CI sureleri optimize edilecek
- Prisma schema dogrulamasi CI'da calisacak
- Anchor build CI ortaminda calisacak

> **Not:** Bu task'ta deployment (Vercel, AWS, Devnet deploy vb.) yapilandirilmaz. Sadece kalite kontrol pipeline'i kurulur. Deployment pipeline'lari ilerideki fazlarda eklenecektir.

### Neden GitHub Actions?

| | GitHub Actions | Jenkins | CircleCI | GitLab CI |
|---|---------------|---------|----------|-----------|

**Neden 3 ayri job?**
- Paralel calisir → toplam sure kisalir
- Bir paketin hatasi digerleri beklemeden raporlanir
- Job bazinda retry mumkun
- Hangi paketin bozuldugu PR'da net gorunur

## Yapilacaklar

### 1. Ana CI Workflow

#### 1.1 `.github/workflows/ci.yml`

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

# Ayni branch'te yeni push gelirse onceki CI'yi iptal et
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

env:
  TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
  TURBO_TEAM: ${{ vars.TURBO_TEAM }}

jobs:
  # ─────────────────────────────────────────────
  # Hangi paketlerin degistigini tespit et
  # ─────────────────────────────────────────────
  changes:
    name: Detect Changes
    runs-on: ubuntu-latest
    outputs:
      frontend: ${{ steps.filter.outputs.frontend }}
      backend: ${{ steps.filter.outputs.backend }}
      contracts: ${{ steps.filter.outputs.contracts }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            frontend:
              - 'frontend/**'
              - 'package.json'
              - 'pnpm-lock.yaml'
              - 'turbo.json'
              - 'tsconfig.json'
            backend:
              - 'backend/**'
              - 'package.json'
              - 'pnpm-lock.yaml'
              - 'turbo.json'
              - 'tsconfig.json'
            contracts:
              - 'contracts/**'

  # ─────────────────────────────────────────────
  # Frontend: lint, type-check, format, build
  # ─────────────────────────────────────────────
  frontend:
    name: Frontend
    runs-on: ubuntu-latest
    needs: changes
    if: needs.changes.outputs.frontend == 'true'
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm --filter @web3-gaming/frontend lint

      - name: Type Check
        run: pnpm --filter @web3-gaming/frontend type-check

      - name: Format Check
        run: pnpm --filter @web3-gaming/frontend format:check

      - name: Build
        run: pnpm --filter @web3-gaming/frontend build

  # ─────────────────────────────────────────────
  # Backend: lint, type-check, format, build
  # ─────────────────────────────────────────────
  backend:
    name: Backend
    runs-on: ubuntu-latest
    needs: changes
    if: needs.changes.outputs.backend == 'true'
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm --filter @web3-gaming/backend lint

      - name: Type Check
        run: pnpm --filter @web3-gaming/backend type-check

      - name: Format Check
        run: pnpm --filter @web3-gaming/backend format:check

      - name: Build
        run: pnpm --filter @web3-gaming/backend build

      - name: Prisma Validate
        run: pnpm --filter @web3-gaming/backend exec prisma validate

      - name: Prisma Generate (dry-run)
        run: pnpm --filter @web3-gaming/backend exec prisma generate

  # ─────────────────────────────────────────────
  # Contracts: cargo clippy, cargo check, anchor build
  # ─────────────────────────────────────────────
  contracts:
    name: Contracts
    runs-on: ubuntu-latest
    needs: changes
    if: needs.changes.outputs.contracts == 'true'
    steps:
      - uses: actions/checkout@v4

      - name: Setup Anchor (Solana CLI + Rust + Anchor)
        uses: metadaoproject/setup-anchor@v2
        with:
          anchor-version: '0.30.1'
          solana-cli-version: '1.18.26'
          node-version: '22'

      - uses: pnpm/action-setup@v4

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Cargo Clippy (Lint)
        working-directory: contracts
        run: cargo clippy --all-targets -- -D warnings

      - name: Cargo Check (Type Check)
        working-directory: contracts
        run: cargo check

      - name: Anchor Build
        working-directory: contracts
        run: anchor build

      - name: Verify IDL Generated
        working-directory: contracts
        run: |
          test -f target/idl/escrow.json && echo "IDL generated successfully" || (echo "IDL not found" && exit 1)

  # ─────────────────────────────────────────────
  # Tum job'larin basarili oldugunu dogrulaayan gate job
  # Branch protection bu job'a bagli olacak
  # ─────────────────────────────────────────────
  ci-gate:
    name: CI Gate
    runs-on: ubuntu-latest
    needs: [changes, frontend, backend, contracts]
    if: always()
    steps:
      - name: Check Results
        run: |
          echo "Frontend needed: ${{ needs.changes.outputs.frontend }}"
          echo "Backend needed: ${{ needs.changes.outputs.backend }}"
          echo "Contracts needed: ${{ needs.changes.outputs.contracts }}"

          # Frontend job kontrolu
          if [[ "${{ needs.changes.outputs.frontend }}" == "true" && "${{ needs.frontend.result }}" != "success" ]]; then
            echo "Frontend CI failed"
            exit 1
          fi

          # Backend job kontrolu
          if [[ "${{ needs.changes.outputs.backend }}" == "true" && "${{ needs.backend.result }}" != "success" ]]; then
            echo "Backend CI failed"
            exit 1
          fi

          # Contracts job kontrolu
          if [[ "${{ needs.changes.outputs.contracts }}" == "true" && "${{ needs.contracts.result }}" != "success" ]]; then
            echo "Contracts CI failed"
            exit 1
          fi

          echo "All required checks passed"
```

**Onemli Notlar:**
- `concurrency` — ayni branch icin birden fazla CI calismaz, yenisi gelince eskiyi iptal eder
- `--frozen-lockfile` — CI'da `pnpm-lock.yaml` degismez, tutarsizlikta hata verir
- `dorny/paths-filter` — sadece degisen paket icin job calistirir (ornek: sadece frontend degistiyse contracts job'u atlanir)
- `ci-gate` — tum job'larin sonucunu tek bir status check'e toplar, branch protection bu job'a baglanir
- `if: always()` — gate job her durumda calisir (skip edilen job'lari da kontrol eder)
- `metadaoproject/setup-anchor@v2` — Solana CLI, Anchor CLI ve Rust toolchain'i tek adimda kurar

### 2. Prisma Migration Kontrolu (Opsiyonel Workflow)

#### 2.1 `.github/workflows/prisma-check.yml`

```yaml
name: Prisma Check

on:
  pull_request:
    paths:
      - 'backend/prisma/**'

jobs:
  prisma:
    name: Prisma Schema Validation
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4

      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Validate Schema
        run: pnpm --filter @web3-gaming/backend exec prisma validate

      - name: Check for pending migrations
        run: |
          pnpm --filter @web3-gaming/backend exec prisma migrate diff \
            --from-migrations ./backend/prisma/migrations \
            --to-schema-datamodel ./backend/prisma/schema.prisma \
            --exit-code || echo "::warning::Schema has changes not covered by migrations"
```

**Not:** Bu workflow sadece `backend/prisma/` altinda degisiklik oldugunda calisir. Schema ile migration'larin senkron olup olmadigini kontrol eder.

### 3. Branch Protection Kurallari

#### 3.1 GitHub Repository Settings

GitHub repository sayfasinda: **Settings → Branches → Branch protection rules → Add rule**

##### `main` branch icin:

| Kural | Deger | Gerekcesi |
|-------|-------|-----------|
| **Branch name pattern** | `main` | Ana branch'i korur |
| **Require a pull request before merging** | Aktif | Direkt push engellenir |
| **Required approvals** | 1 (baslangiç icin) | En az 1 kisi code review yapmali |
| **Dismiss stale pull request approvals** | Aktif | Yeni commit gelince onceki approval gecersiz olur |
| **Require status checks to pass** | Aktif | CI gecmeden merge yapilamaz |
| **Required status checks** | `CI Gate` | Ana CI workflow'unun gate job'u |
| **Require branches to be up to date** | Aktif | Merge oncesi main ile guncel olmali |
| **Require conversation resolution** | Aktif | Tum review yorumlari cozulmeli |
| **Do not allow bypassing** | Aktif | Admin'ler bile kurallari atlayamaz |

##### `develop` branch icin (eger olusturulursa):

| Kural | Deger |
|-------|-------|
| **Branch name pattern** | `develop` |
| **Require status checks to pass** | `CI Gate` |
| **Require branches to be up to date** | Aktif |

**Notlar:**
- Branch protection kurallari GitHub UI veya API uzerinden tanimlanir — workflow dosyasiyla yapilandirilmaz
- `CI Gate` job ismi branch protection icinde status check olarak secilir
- Bu kurallar ilk CI calistiktan sonra aktif edilebilir (cunku status check'in bir kez calisip gorunmesi gerekir)

### 4. Pre-commit Hook'lar (Husky + lint-staged)

CI pipeline PR acildiginda calisir. Ancak gelisitricinin hatalari daha erken — commit aninda — yakalamasi icin yerel pre-commit hook'lar da kurulmalidir.

#### 4.1 Bagimliliklari Kur

```bash
# Root dizinde
pnpm add -D husky lint-staged
```

#### 4.2 Husky'yi Baslat

```bash
pnpm exec husky init
```

Bu komut:
- `.husky/` dizinini olusturur
- `package.json`'a `"prepare": "husky"` script'ini ekler

#### 4.3 `.husky/pre-commit`

```bash
pnpm exec lint-staged
```

#### 4.4 Root `package.json` — lint-staged Yapilandirmasi

`package.json` icine eklenecek:

```json
{
  "lint-staged": {
    "frontend/**/*.{ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "backend/**/*.{ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "frontend/**/*.{json,css,md}": [
      "prettier --write"
    ],
    "backend/**/*.{json,css,md}": [
      "prettier --write"
    ]
  }
}
```

**Notlar:**
- `lint-staged` sadece staged dosyalar uzerinde calisir — tum projeyi taramaz, hizlidir
- `--fix` ile otomatik duzeltme yapar, duzeltilmis dosyayi tekrar stage'e ekler
- Rust/Anchor dosyalari icin lint-staged kullanilmaz — `cargo clippy` tum crate'i tarar, tek dosya bazinda calistirilmaz
- Pre-commit hook basarisiz olursa commit engellenir — gelistirici hatalarini commit oncesi gorur

### 5. GitHub Actions Cache Stratejisi

CI surelerini kısaltmak icin uc katmanli cache stratejisi uygulanir:

#### 5.1 Cache Katmanlari

| Katman | Ne Cache'lenir | Nasil | Cache Key |
|--------|---------------|-------|-----------|
| **pnpm store** | Indirilen npm paketleri | `actions/setup-node` icindeki `cache: 'pnpm'` | `pnpm-lock.yaml` hash'i |
| **Turborepo cache** | Build ciktilari, lint sonuclari | Turbo'nun dahili cache mekanizmasi | Task hash'leri |
| **Rust target** | Derlenmis Rust crate'leri | `actions/cache` ile `contracts/target/` | `Cargo.lock` hash'i |

#### 5.2 Rust Cache (Contracts Job Icine Eklenecek)

```yaml
      - name: Cache Rust dependencies
        uses: actions/cache@v4
        with:
          path: |
            ~/.cargo/bin/
            ~/.cargo/registry/index/
            ~/.cargo/registry/cache/
            ~/.cargo/git/db/
            contracts/target/
          key: rust-${{ runner.os }}-${{ hashFiles('contracts/Cargo.lock') }}
          restore-keys: |
            rust-${{ runner.os }}-
```

**Not:** Rust derleme suresi uzun oldugu icin `target/` dizininin cache'lenmesi CI suresini onemli olcude kisaltir. Ilk calistirmada ~3-5 dk suren Anchor build, cache ile ~30-60 saniyeye duser.

#### 5.3 Turborepo Remote Cache (Opsiyonel)

Turborepo remote cache, farkli CI calistirmalari arasinda build sonuclarini paylasir:

1. Vercel hesabi ile Turborepo remote cache'i aktif et
2. GitHub repository'de secret'lari tanimla:
   - `TURBO_TOKEN` — Vercel'den alinan token
   - `TURBO_TEAM` — Vercel team slug'i (variable olarak)
3. Workflow'daki `env` blogu bu degerleri otomatik kullanir

> Remote cache opsiyoneldir. Kucuk ekiplerde lokal cache yeterlidir. Ekip buyudukce veya CI sureleri uzadikca aktif edilir.
