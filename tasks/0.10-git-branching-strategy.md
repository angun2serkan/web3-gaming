# Task 0.10 — Git Branching Stratejisi

**Faz:** 0 — Proje Altyapisi
**Tahmini Efor:** 0.5 gun
**Durum:** Beklemede
**Bagimlilik:** Task 0.6 (CI/CD pipeline) tamamlanmis olmali

---

## Amac

Projenin git branch yapisini, merge kurallarini, branch protection ayarlarini ve PR workflow'unu olusturmak. Bu task tamamlandiginda:

- `main` ve `develop` branch'leri rolleri tanimlanmis olacak
- Branch isimlendirme konvansiyonu belirlenmis olacak
- GitHub branch protection rules aktif olacak
- PR review ve merge sureci netlestirilmis olacak
- Release workflow'u dokumante edilmis olacak

## Adimlar

### Adim 1 — `develop` Branch'ini Olustur

`main` branch'i production-ready kodu temsil eder. Gunluk gelistirme `develop` uzerinden yapilir.

```bash
# main branch'inden develop olustur
git checkout main
git pull origin main
git checkout -b develop
git push -u origin develop
```

**Branch rolleri:**

| Branch | Rol | Kim merge eder | CI zorunlu mu? |
|--------|-----|----------------|----------------|
| `main` | Production-ready kod. Her zaman deploy edilebilir durumda. | Sadece `develop` veya `release/*` branch'inden PR ile | Evet |
| `develop` | Aktif gelistirme branch'i. Feature'lar buraya merge edilir. | Feature/fix PR'lari ile | Evet |

### Adim 2 — Branch Isimlendirme Konvansiyonunu Tanimla

Tum branch isimleri kucuk harf ve kebab-case kullanir.

| Prefix | Kullanim | Ornek |
|--------|----------|-------|
| `feature/` | Yeni ozellik gelistirme | `feature/lobby-ui` |
| `fix/` | Bug duzeltme | `fix/socket-disconnect-handling` |
| `refactor/` | Kod iyilestirme (davranis degisikligi yok) | `refactor/game-service-cleanup` |
| `release/` | Release hazirligi (son testler, versiyon guncelleme) | `release/1.0.0` |
| `hotfix/` | Production'daki kritik bug icin acil duzeltme | `hotfix/escrow-overflow` |
| `chore/` | CI, config, dependency, dokumantasyon gibi destekleyici isler | `chore/update-dependencies` |

**Kurallar:**
- Branch ismi ilgili task numarasini icerebilir: `feature/0.10-branching-strategy`
- Cok uzun isimlerden kacin — 3-5 kelime ideal
- Bos karakter yerine tire (`-`) kullan

### Adim 3 — GitHub Branch Protection Rules Ayarla

GitHub repo sayfasindan: **Settings → Branches → Add branch protection rule**

#### 3a — `main` branch korumasi

| Ayar | Deger |
|------|-------|
| **Branch name pattern** | `main` |
| **Require a pull request before merging** | Acik |
| — Required approving reviews | `1` (takim buyuyunce `2` yap) |
| — Dismiss stale pull request approvals when new commits are pushed | Acik |
| **Require status checks to pass before merging** | Acik |
| — Status checks that are required | `CI Gate` |
| **Require branches to be up to date before merging** | Acik |
| **Require conversation resolution before merging** | Acik |
| **Do not allow bypassing the above settings** | Acik (admin dahil) |
| **Restrict who can push to matching branches** | Acik — sadece PR ile merge |
| **Allow force pushes** | KAPALI |
| **Allow deletions** | KAPALI |

#### 3b — `develop` branch korumasi

| Ayar | Deger |
|------|-------|
| **Branch name pattern** | `develop` |
| **Require a pull request before merging** | Acik |
| — Required approving reviews | `1` |
| **Require status checks to pass before merging** | Acik |
| — Status checks that are required | `CI Gate` |
| **Allow force pushes** | KAPALI |
| **Allow deletions** | KAPALI |

> **Not:** Bu ayarlar GitHub Free plan'da sinirli olabilir. `Require approving reviews` ozeligi public repo'larda veya GitHub Team/Enterprise plan'da tam calisiyor. Solo gelistirmede review sayisini `0` birak ama diger korumalar aktif olsun.

### Adim 6 — Merge Stratejisini Belirle

| Durum | Merge Yontemi | Neden |
|-------|---------------|-------|
| `feature/*` → `develop` | **Squash and merge** | Feature'in tum commit'lerini tek commit'e sikitirir, develop gecmisi temiz kalir |
| `fix/*` → `develop` | **Squash and merge** | Ayni sebep |
| `develop` → `main` | **Merge commit** (no squash) | Release gecmisi korunur, hangi feature'larin dahil oldugu gorulur |
| `release/*` → `main` | **Merge commit** | Release icerigi net kalir |
| `hotfix/*` → `main` | **Merge commit** | Sonrasinda `main` → `develop`'a da merge edilir |

**GitHub'da ayarlama:**
Settings → General → Pull Requests:
- [x] Allow squash merging (default commit message: PR title)
- [x] Allow merge commits
- [ ] Allow rebase merging (KAPALI — gecmis karisikligini onler)

### Adim 7 — Git Workflow'lari Dokumante Et

#### 7a — Standart feature gelistirme akisi

```bash
# 1. develop'dan yeni branch olustur
git checkout develop
git pull origin develop
git checkout -b feature/lobby-ui

# 2. Gelistirme yap, commit'le
git add <dosyalar>
git commit -m "feat: implement lobby page layout"

# 3. Push et ve PR ac
git push -u origin feature/lobby-ui
# GitHub'da develop'a PR ac

# 4. CI gecsin, review alinsin, squash merge yapilsin
# 5. Merge sonrasi feature branch'i sil
```

#### 7b — Hotfix akisi

```bash
# 1. main'den hotfix branch'i ac
git checkout main
git pull origin main
git checkout -b hotfix/escrow-overflow

# 2. Fix yap, commit'le
git add <dosyalar>
git commit -m "fix: prevent arithmetic overflow in escrow calculation"

# 3. main'e PR ac ve merge et
git push -u origin hotfix/escrow-overflow
# GitHub'da main'e PR ac, merge commit ile merge et

# 4. main'deki fix'i develop'a da al
git checkout develop
git pull origin develop
git merge origin/main
git push origin develop
```

#### 7c — Release akisi

```bash
# 1. develop'dan release branch'i ac
git checkout develop
git pull origin develop
git checkout -b release/1.0.0

# 2. Son hazirliklari yap (versiyon numarasi, changelog, vb.)
git commit -m "chore: bump version to 1.0.0"

# 3. main'e PR ac ve merge et (merge commit)
git push -u origin release/1.0.0
# GitHub'da main'e PR ac

# 4. Merge sonrasi main'den tag olustur
git checkout main
git pull origin main
git tag -a v1.0.0 -m "Release 1.0.0"
git push origin v1.0.0

# 5. main'i develop'a merge et (release commit'leri icin)
git checkout develop
git merge origin/main
git push origin develop
```

### Adim 8 — Commit Mesaji Konvansiyonu

[Conventional Commits](https://www.conventionalcommits.org/) standardini kullan:

```
<tip>(<kapsam>): <aciklama>
```

| Tip | Kullanim |
|-----|----------|
| `feat` | Yeni ozellik |
| `fix` | Bug duzeltme |
| `refactor` | Kod iyilestirme (davranis degisikligi yok) |
| `chore` | Build, CI, dependency, config degisiklikleri |
| `docs` | Sadece dokumantasyon degisikligi |
| `test` | Test ekleme veya guncelleme |
| `style` | Kod formatlama (davranisi etkilemeyen) |

**Kapsam ornekleri:** `frontend`, `backend`, `contracts`, `ci`, `prisma`

**Ornekler:**
```
feat(frontend): add lobby page with room list
fix(backend): handle socket disconnect during game
refactor(backend): extract game logic to service layer
chore(ci): add Prisma schema validation step
docs: update CLAUDE.md with branching strategy
```

**Kurallar:**
- Kucuk harfle basla
- Sonuna nokta koyma
- Imperative mood kullan (Turkce'de mastar fiil: "add", "fix", "update")
- Ilk satir 72 karakteri gecmesin
- Gerekirse bos satir birakip body ekle

### Adim 9 — CLAUDE.md'ye Branching Kurallarini Ekle

Root `CLAUDE.md` dosyasina su kurallari ekle:

```markdown
## Git Workflow

- Default branch: `develop` — tum feature PR'lari buraya acilir
- `main` branch'i production-ready — sadece `develop`, `release/*`, `hotfix/*`'ten merge alir
- Branch isimlendirme: `feature/`, `fix/`, `refactor/`, `release/`, `hotfix/`, `chore/` prefix'leri
- Commit mesaji: Conventional Commits (`feat(frontend): add lobby page`)
- Feature/fix → develop: squash merge. Develop → main: merge commit.
- IMPORTANT: `main` branch'ine dogrudan push YAPMA — her zaman PR ac
```

### Adim 10 — Dogrulama

Tum adimlarin dogru uygulandigini kontrol et:

```bash
# develop branch'i mevcut mu?
git branch -a | grep develop

# Remote'ta develop var mi?
git ls-remote --heads origin develop

# Default branch develop mi? (GitHub API ile kontrol)
gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'

# Branch protection aktif mi?
gh api repos/{owner}/{repo}/branches/main/protection --jq '.required_status_checks.contexts'
gh api repos/{owner}/{repo}/branches/develop/protection --jq '.required_status_checks.contexts'
```

**Manuel kontrol listesi:**

- [ ] `develop` branch'i olusturuldu ve push edildi
- [ ] GitHub default branch `develop` olarak degistirildi
- [ ] `main` branch protection aktif (PR zorunlu, CI Gate zorunlu, force push kapali)

- [ ] PR template guncellendi (degisiklik tipi ve branch kontrol maddesi eklendi)
