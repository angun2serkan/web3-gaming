## Amac

Frontend ve backend paketlerinde oyun mekanigiyle, oda sistemiyle, oyuncu yonetimiyle ve WebSocket iletisimiyle ilgili TypeScript tip tanimlarini ve sabit degerlerini olusturmak. Her paket kendi tiplerini **kendi icinde bagimsiz olarak** tanimlar — paylasilan (shared) bir paket veya dizin **kullanilmaz**.

Bu task tamamlandiginda:
- `frontend/src/types/` altinda frontend'in ihtiyac duydugu tum tipler tanimlanmis olacak
- `backend/src/types/` altinda backend'in ihtiyac duydugu tum tipler tanimlanmis olacak
- `frontend/src/constants/` ve `backend/src/constants/` altinda ortak sabitler (Socket event isimleri, oyun konfigurasyonu) tanimlanmis olacak
- Socket.io event isimleri ve payload yapilari her iki tarafta tutarli olacak
- Prisma schema'daki model ve enum'lar backend tiplerinin temelini olusturacak
- Ilerideki sprint'lerde (Faz 1 Sprint 1-4) yazilacak is mantigi bu tipleri kullanacak

> **Not:** Bu task'ta is mantigi yazilmaz. Sadece tip tanimlari, enum'lar, interface'ler ve sabitler olusturulur.

## Mimari Karar: Neden `shared/` Paketi Yok?

Orijinal roadmap `shared/` altinda ortak TypeScript tipleri oneriyordu. Bu yaklasim **reddedildi**. Tipler her pakette bagimsiz olarak tanimlanacak.

### Karsilastirma

| | Shared Paket (`shared/`) | Bagimsiz Tipler (Her Pakette Ayri) |
|---|--------------------------|-----------------------------------|
| **Bagimlilik** | Frontend ve backend `shared` paketine bagimli — bir degisiklik her iki paketi etkiler | Paketler birbirinden bagimsiz — biri degisirken digeri etkilenmez |
| **Esneklik** | Frontend ve backend ayni yapiyi kullanmak zorunda — API yaniti ile UI state'i karisir | Frontend sadece ihtiyac duydugu alanlari tanimlar, backend kendi ihtiyaclarina gore genisletir |
| **Ayristirma** | Paketleri ayri repo'lara tasimak zor — `shared` bagimliligi koparilmali | Her paket bagimsiz — ileride ayri repo'ya tasinabilir |
| **Build karmasikligi** | `shared` degisince her iki paket yeniden build edilmeli | Sadece degisen paket build edilir |
| **Prisma entegrasyonu** | Prisma tipleri `shared`'a mi backend'e mi ait? Belirsizlik | Prisma tipleri backend'de kalir — dogal sahiplik |
| **Tip farklilasmasi** | Frontend'in ihtiyac duymadigi alanlar (orn. `createdAt`, `updatedAt`) zorunlu olur | Frontend sadece gosterdigi alanlari tanimlar |

### Gercek Dunya Ornegi

Backend'teki `Room` tipi:
```typescript
// Backend — veritabani modeliyle birebir eslesir
interface Room {
  id: string;
  type: RoomType;
  entryFee: Decimal;       // Prisma Decimal tipi
  status: RoomStatus;
  minPlayers: number;
  maxPlayers: number | null;
  createdAt: Date;
  closedAt: Date | null;
  players: RoomPlayer[];   // Iliskisel veri
  games: Game[];           // Iliskisel veri
}
```

Frontend'teki `Room` tipi:
```typescript
// Frontend — sadece UI'in gosterdigi alanlar
interface Room {
  id: string;
  type: RoomType;
  entryFee: number;        // number yeterli (Decimal gereksiz)
  status: RoomStatus;
  playerCount: number;     // players[].length degil, sayi yeterli
  maxPlayers: number | null;
  poolSize: number;        // Hesaplanmis alan — backend'de yok
}
```

> **Sonuc:** Frontend ve backend farkli ihtiyaclara sahiptir. Ortak paket iki tarafin da ihtiyaclarini karsilamaya calisir ve gereksiz karmasiklik yaratir. Her paketin kendi tiplerini tanimlamasi daha temiz, daha esnek ve daha surdurulebilirdir.

### Senkronizasyon Nasil Saglanir?

Tipler ayriysa iki tarafin uyumsuz hale gelmesi riski vardir. Bu risk su sekilde yonetilir:

| Yontem | Aciklama |
|--------|----------|
| **Socket event isimleri** | Her iki tarafta ayni string sabitler kullanilir. CI'da grep ile kontrol edilebilir |
| **API kontrati** | Backend'in gonderdigi payload yapisi dokuimante edilir. Frontend buna gore tip tanimlar |
| **Prisma kaynak** | Backend tipleri Prisma schema'dan turetilir. Frontend tipleri backend'in API yanit yapisina gore sekillanir |
| **Code review** | Socket event veya API yapisi degistiginde her iki tarafin guncellenmesi PR review'da kontrol edilir |

#### 1.1 `backend/src/types/enums.ts`

Prisma schema'daki enum'larin TypeScript karsiliklari. Prisma Client kendi enum tiplerini uretir, ancak bu enum'lar is mantigi katmaninda da (Prisma'dan bagimsiz) kullanilacagi icin ayrica tanimlanir.

  IN_PROGRESS = 'IN_PROGRESS',

  SELECTION = 'SELECTION',
  PROTECTED = 'PROTECTED',
  PIE_FORM = 'PIE_FORM',

**Notlar:**
- `RoundPhase` enum'u Prisma schema'da yok — Format A'nin round state machine'i icin ekleniyor (roadmap Task 2.1)
- String enum kullanildi — JSON serialization'da okunabilirlik saglar
- Prisma Client'in urettigi enum'lar da mevcuttur, ancak servis katmaninda Prisma'ya bagimlilik azaltmak icin ayri enum tanimlanir

// ─────────────────────────────────────────────

export interface ClientToServerEvents {
  'lobby:subscribe': () => void;
  'lobby:unsubscribe': () => void;
  'room:create': (data: { type: string; entryFee: number; maxPlayers?: number }) => void;
  'room:join': (data: { roomId: string }) => void;
  'room:leave': (data: { roomId: string }) => void;
  'game:select-number': (data: { gameId: string; number: number }) => void;
  'game:player-input': (data: { direction: 'left' | 'right' }) => void;
}

// ─────────────────────────────────────────────
// Server → Client Event'leri (on)
// ─────────────────────────────────────────────

export interface ServerToClientEvents {
  // Lobi
  'lobby:rooms-update': (data: { rooms: LobbyRoom[] }) => void;

  // Oda
  'room:state': (data: { room: RoomState }) => void;
  'room:player-joined': (data: {
    roomId: string;
    playerId: string;
    walletAddress: string;
    playerCount: number;
    poolSize: number;
  }) => void;
  'room:player-left': (data: {
    roomId: string;
    playerId: string;
    playerCount: number;
    poolSize: number;
  }) => void;
  'room:countdown': (data: { roomId: string; remainingMs: number }) => void;
  'room:cancelled': (data: { roomId: string; reason: string }) => void;

  // Format A
  'game:started': (data: {
    gameId: string;
    format: 'FORMAT_A' | 'FORMAT_B';
    playerCount: number;
    poolAmount: number;
  }) => void;
  'game:selection-phase': (data: {
    gameId: string;
    roundNumber: number;
    numberRange: { min: number; max: number };
    durationMs: number;
    endsAt: number;
  }) => void;
  'game:protected-reveal': (data: ProtectedReveal & { gameId: string; roundNumber: number }) => void;
  'game:pie-formed': (data: { gameId: string; roundNumber: number; slices: PieSlice[] }) => void;
  'game:arrow-spin': (data: ArrowSpin & { gameId: string; roundNumber: number }) => void;
  'game:elimination': (data: Elimination & { gameId: string; roundNumber: number }) => void;
  'game:round-end': (data: {
    gameId: string;
    roundNumber: number;
    remainingPlayerCount: number;
    nextNumberRange: { min: number; max: number } | null;
  }) => void;
  'game:winner': (data: FormatAWinner) => void;

  // Format B
  'game:state-update': (data: {
    players: PlayerSphere[];
    snake: SnakeVisual;
    waterLevel: number;
  }) => void;
  'game:player-eliminated': (data: FormatBElimination) => void;
  'game:safe-zone-update': (data: { safeZones: SafeZoneVisual[] }) => void;
  'game:format-b-winner': (data: FormatBResult) => void;

  // Genel
  'error': (data: { code: string; message: string }) => void;
}

## Onemli Notlar

### Frontend vs Backend Tip Farkliliklari

| Alan | Backend | Frontend | Gerekcesi |
|------|---------|----------|-----------|
| `entryFee` | `number` (Prisma Decimal'den donusturulmus) | `number` | Ayni — iki tarafta da number |
| `createdAt` | `Date` | Yok (gerekmedikce) | Frontend tarih alanlarini nadiren gosterir |
| `players` | `RoomPlayer[]` (tam obje) | `RoomPlayerInfo[]` (sadece gorunur alanlar) | Frontend iliskisel derinlige ihtiyac duymaz |
| `poolAmount` | `number` | `poolSize: number` | Frontend icin daha okunabilir isim |
| Yilan `glowIntensity` | Yok | `number` | Sadece gorsel — backend ilgilenmez |
| `isMe` (kazanan) | Yok | `boolean` | UI'da "sen kazandin" gosterimi icin |
| `profitLoss` | Yok | `number` | Hesaplanmis UI alani |

### Senkronizasyon Stratejisi

Tipler bagimsiz oldugu icin iki tarafin uyumsuz hale gelmemesi icin:

1. **Socket event isimleri** — `SOCKET_EVENTS` sabiti her iki tarafta ayni degerlerle tanimli. Deger degistiginde her iki taraf da guncellenmeli.
2. **Payload yapisi** — Backend `socket.ts`'deki payload tipleri "kaynak" olarak kabul edilir. Frontend kendi tiplerini bu yapiya gore modeller.
3. **Code review kurali** — `socket.ts` veya `constants/` degisikligi iceren PR'larda her iki tarafin da kontrol edilmesi zorunlu.
4. **Ileride (opsiyonel)** — CI'da basit bir script ile frontend ve backend `SOCKET_EVENTS` degerlerinin eslestigini dogrulamak mumkun.
